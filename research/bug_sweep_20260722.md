## Cross-Cutting Patterns

### Pattern A: `_FakeAsOf` ‚Äî 5 copies, same bugs
A 100-line fake as-of store is copy-pasted across **5 test files** (`test_parity.py`, `test_ml_factors.py`, `test_baseline_factors.py`, `test_label001_integration.py`, `test_split001_integration.py`). Each copy has the same bugs:
- `latest_available` only uses `keys[0]` ‚Äî multi-key queries return wrong data
- `as_of()` is a stub returning no real data ‚Äî masks production store behavior
- Unrequested field fallback can return wrong columns
- `int(timestamp() * 1e6)` truncation can lose one microsecond
- Parameter name mismatch with protocol (`instant` ‚âÝ `decision_time`)

Any fix must be replicated 5√ó. No shared `conftest.py` or `tests/helpers/` module exists.

### Pattern B: `_require_utc` ‚Äî duplicated in 4 source modules
Identical UTC-validation logic copy-pasted in `ml.py`, `labels.py`, `composite.py`, `baseline.py`, plus `parity.py` (SERV-001). Each raises a different exception type (`MLFactorError`, `LabelError`, etc.). Should be a shared utility.

### Pattern C: Test helpers imported via private (`_`) names
Multiple test files import `_DATASET`, `_INSTRUMENT_DATASET_ID`, `_build_catalog_asof` from `test_baseline_factors.py` ‚Äî all private names. Renaming or restructuring that file silently breaks all downstream test imports.

### Pattern D: `availability_time = decision_time` instead of `period_start + 1d`
BASE-001, COMP-001, and ML-001 all set `availability_time=decision_time` on `FactorValue`. The BASE-001 ticket explicitly requires `period_start + 1d`. This violates BAR-001 timestamp semantics and misleads downstream consumers that rely on `availability_time` for point-in-time eligibility.

### Pattern E: Float precision via `int(timestamp() * 1_000_000)` ‚Äî 5+ locations
`parity.py:102`, `test_parity.py:45`, `test_ml_factors.py:42`, `test_baseline_factors.py:60`, `as_of.py:66` all convert `datetime ‚Üí float ‚Üí int` for microsecond conversion. For timestamps post-2022, float64's 53-bit mantissa cannot represent all microsecond values exactly, resulting in off-by-one-microsecond errors. Should use integer `divmod` arithmetic (as in `bars.py:294-298`).
