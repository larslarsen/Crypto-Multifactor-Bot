# CAT-001 — Initialize the control catalog and migration runner

**Priority:** P0  
**Dependencies:** repository bootstrap only  
**Layer:** catalog
**Status:** SUPERSEDED BY CAT-001A (ACCEPTED)
**Superseded by:** REVIEW-0002_CAT-001A_FINAL.md (REVIEW-0001 records CAT-001 superseded; CAT-001A accepted and brings CAT-001 into conformance)

## Objective

Create a deterministic, forward-only SQLite migration system and initialize the control catalog from the approved baseline schema.

## Deliverables

- migration directory and ordered migration discovery;
- `cf catalog init --database <path>`;
- `cf catalog status --database <path>`;
- migration-history table with filename and SHA-256;
- transactional migration application;
- foreign keys enabled on every connection;
- configurable busy timeout and WAL mode for local use;
- clear failure on changed checksum, gap, duplicate version, or failed statement;
- tests using temporary databases.

## Invariants

- Applying migrations twice is idempotent.
- A migration already recorded with a different hash aborts.
- No migration is edited after acceptance; corrections use a later migration.
- A failed migration does not record success.
- SQLite stores control metadata only.
- Database path is explicit or configuration-derived, never hard-coded.

## Required tests

1. new database reaches latest version;
2. second application makes no changes;
3. altered applied migration is rejected;
4. invalid migration rolls back;
5. foreign-key violation is rejected;
6. concurrent/read behavior is documented and minimally exercised;
7. status command reports pending/applied migrations.

## Acceptance commands

```bash
pytest -q tests/catalog
ruff check src tests
mypy src
cf catalog init --database .local/control.db
cf catalog status --database .local/control.db
```


## Agent stop condition

After the acceptance commands pass, produce a change report and stop. Do not begin the next ticket.
